# PPT Agent General Workflow and Rules

This document defines a reusable workflow for agents that create or revise PowerPoint decks. It extracts general rules from prior PPT editing work and removes project-specific assumptions.

## 1. Protect User Edits

- Always assume the user may have manually edited the deck after the last agent run.
- Inspect the current PPT file before changing any slide.
- Only edit the requested slide range.
- If the user marks a slide or element as manually adjusted, do not rebuild or overwrite it.
- Prefer narrow update scripts for specific slides or slide groups.
- Do not rerun old broad generation scripts unless the user explicitly asks for a full rebuild.
- Avoid changing unrelated metadata, slide order, theme settings, or master layouts.

## 2. Start From the Current Deck

For every request:

1. Open or inspect the current PPT.
2. Identify target slides and existing shapes.
3. Check nearby slides for title style, margins, fonts, visual density, and layout conventions.
4. Determine which objects are user-authored, template-provided, or generated by prior scripts.
5. Make edits against the live deck state, not against memory or old script assumptions.

## 3. Define One Message Per Slide

Each slide should communicate one main point.

Good slide content:

- short noun phrases;
- key numbers;
- method or workflow keywords;
- result labels;
- concise comparison terms.

Avoid:

- paragraphs;
- full explanations;
- long copied captions;
- redundant text that repeats what the speaker will say;
- multiple competing conclusions on one slide.

Speaker notes or narration can contain the explanation. Slide text should support scanning.

## 4. Use a Consistent Layout Grammar

Choose a simple layout and apply it consistently:

- title plus text on top, figure or table below;
- title plus text on left, figure or table on right;
- one main visual for workflow or architecture slides;
- two aligned visuals only when comparison is the point;
- table-centered layout for quantitative summaries.

Avoid mixing many unrelated text boxes, callouts, figures, and decorative shapes on one slide.

For result slides:

- make the figure or table the visual center;
- keep bullets to 2-3 keyword lines;
- align figures by top edge and height;
- align captions by baseline;
- keep table row heights and column alignment consistent.

## 5. Typography Rules

- Match the deck's existing template font family and title style.
- Use readable projection sizes: main bullets usually around 24-30 pt.
- Use sub-bullets sparingly; keep them large enough to read, usually 20-24 pt.
- Keep captions and table text at least 18-20 pt whenever possible.
- Avoid tiny labels unless they are part of an imported figure that cannot be enlarged.
- Do not let text overlap with figures, legends, footers, page numbers, or template elements.
- Do not rely only on estimated PPT coordinates. Always verify rendered screenshots when precision matters.

## 6. Visual Asset Rules

- Use original project, paper, product, or domain figures when available and appropriate.
- Preserve the meaning of source figures. Do not redraw them in a way that changes the underlying representation.
- Crop excessive white margins before inserting images.
- Use high-resolution images when available.
- Keep aspect ratios intact unless the source is explicitly decorative.
- Redraw diagrams only when the original is missing, unreadable, inconsistent, or too low-resolution.
- Store generated or derived image assets in a clear local asset folder.
- Name generated assets descriptively so later agents can understand their purpose.

## 7. Table Rules

- Build tables directly in PowerPoint when readability matters.
- Avoid using screenshot tables unless the table is simple and already legible.
- Use a clear header style with sufficient contrast.
- Highlight only rows or cells that support the slide's main message.
- Keep column widths, row heights, and alignment consistent.
- Use concise labels; move long explanations to narration or notes.
- Verify table text size in the rendered slide.

## 8. Caption Rules

- Captions should be short, factual, and useful.
- Use captions to identify figures, conditions, datasets, methods, or metrics.
- Avoid long figure captions copied from papers or reports.
- If the user removed a caption, do not add it back without reason.
- Keep caption placement consistent across slides.

Good caption style:

```text
Model workflow
Input feature map
Accuracy by subject
Average latency
Channel ranking
```

## 9. Visual Balance

- Keep margins balanced.
- Avoid large unused blank regions unless intentional.
- Do not fill empty space with decorative elements.
- Keep visual hierarchy clear: title, key text, main visual, secondary labels.
- If two figures are in one row, align top edges and use comparable heights.
- If two figures are stacked, align widths and left edges.
- Keep legends, labels, and annotations close to the visual they explain.
- Avoid heavy callout boxes unless they summarize one essential takeaway.

## 10. Accuracy Rules

- Do not simplify technical, scientific, financial, legal, medical, or product content beyond what is correct.
- Preserve metric names, units, axis labels, dataset names, and method relationships.
- Confirm whether a figure is illustrative, experimental, conceptual, or quantitative.
- Do not invent numbers, labels, or claims to improve visual symmetry.
- If source material is ambiguous, keep wording conservative or ask the user.
- Make sure rewritten slide text still matches the referenced paper, report, dataset, or source material.

## 11. Editing Workflow

Use this process for each PPT editing request:

1. Inspect the current PPT and target slide shapes.
2. Review nearby slides for style and layout conventions.
3. Identify the one message of each target slide.
4. Choose the layout before editing.
5. Prepare or crop required images.
6. Write a narrow script that changes only the requested slides.
7. Run the script and save to the intended output file.
8. Export screenshots of changed slides.
9. Ask a separate sub-agent to inspect the exported screenshots against the request or reference image before accepting the result. This independent check helps reduce confirmation bias from the editing agent.
10. Visually inspect screenshots for overlap, alignment, small text, cropping, blank space, and slide-to-slide consistency.
11. Iterate until both the independent check and the local inspection are satisfactory.
12. Run a structure check for small text, accidental manual page numbers, and unexpected shape changes.
13. Summarize exactly what changed and what was verified.

## 12. Automation Rules

- Prefer scripts for repeatable edits, especially when adjusting many shapes.
- Keep scripts scoped to the current task.
- Name scripts by target slide or purpose.
- Do not keep using stale scripts after the deck has changed manually.
- Read shape IDs, names, text, and geometry before editing.
- Use structured PPT APIs where possible instead of ad hoc file manipulation.
- Preserve existing theme, master, and template elements unless the user requests a design change.
- Keep generated artifacts organized so future edits can trace which files are source, derived, or temporary.

## 13. Verification Checks

A PPT change is not complete until rendered output has been checked.

Minimum checks:

- changed slides export successfully;
- exported screenshots are checked by a separate sub-agent when visual fidelity matters;
- no text overlaps with figures or template elements;
- no unexpected tiny text;
- figures are not distorted or incorrectly cropped;
- tables are readable;
- captions align consistently;
- slide numbers or footer elements are not duplicated manually;
- changed slides match nearby deck style.

Useful structural check pattern:

```bash
python - <<'PY'
from pptx import Presentation
import re

pptx_path = 'deck.pptx'
prs = Presentation(pptx_path)

for si, slide in enumerate(prs.slides, start=1):
    small = []
    manual_page_numbers = []

    for i, shape in enumerate(slide.shapes):
        if getattr(shape, 'has_text_frame', False):
            text = shape.text.strip()
            if re.fullmatch(r'\d+\s*/\s*\d+', text):
                manual_page_numbers.append((i, text))

            for paragraph in shape.text_frame.paragraphs:
                for run in paragraph.runs:
                    if run.text.strip() and run.font.size is not None and run.font.size.pt < 18:
                        small.append((i, run.font.size.pt, run.text))

        if getattr(shape, 'has_table', False):
            for ri, row in enumerate(shape.table.rows):
                for ci, cell in enumerate(row.cells):
                    for paragraph in cell.text_frame.paragraphs:
                        for run in paragraph.runs:
                            if run.text.strip() and run.font.size is not None and run.font.size.pt < 18:
                                small.append((f'table {ri},{ci}', run.font.size.pt, run.text))

    if small:
        print('small text:', si, small)
    if manual_page_numbers:
        print('manual page numbers:', si, manual_page_numbers)
PY
```

## 14. Final Delivery Checklist

Before reporting completion:

- confirm which PPT file was edited or created;
- list changed slide numbers;
- mention any generated assets or scripts;
- state what verification was run;
- call out any unresolved risk, such as unavailable source images or inability to export screenshots.

## 15. Core Principle

Reliable PPT work follows this loop:

```text
inspect current deck -> edit narrowly -> export screenshots -> inspect visually -> iterate
```

Good PPT agent work is usually less about adding design elements and more about preserving user edits, simplifying slide content, aligning visual structure, and verifying the rendered result.
